## Concept alignment
- **Concept doc(s)**: [`concepts/permissions/02-auth-request.md`](../../../concepts/permissions/02-auth-request.md) §"AR lifecycle — Submitted → Approved → Revoked / Denied" + [`concepts/permissions/07-templates-and-tools.md`](../../../concepts/permissions/07-templates-and-tools.md) §"adopting a template"
- **Concept claim (verbatim or close paraphrase)**: An org's adoption of a template is expressed via an AR. Over time an org may adopt, revoke, re-adopt — multiple ARs for the same (org, kind) exist in history. The "current" adoption state is the most-recent AR's state.
- **Contradiction**: None — `find_adoption_ar` returning the most-recent AR matches the concept's "current" semantics. Plan text implied a 1-to-1 constraint that concept docs do not themselves impose.
- **Classification**: `concept-silent-plan-filled-gap` (plan was tighter than concept; shipped code matches concept)
- **phi-core leverage status**: `N/A — no phi-core overlap`

## Plan vs. reality
- **Plan said** (plan-archive line 1321, §Part 1 G1 + ADR-0030 D1): *"Template node uniqueness via UNIQUE(kind); adoption carried by AR `provenance_template`."* Implied one AR per (org, kind).
- **Reality (shipped state at current HEAD)**: Migration 0005's `UNIQUE(kind)` is on the **Template** node (platform-level), not on adoption ARs. An org can span multiple ARs per kind over its lifetime. `find_adoption_ar` at [`templates/mod.rs:173-184`](../../../../../../modules/crates/server/src/platform/templates/mod.rs#L173-L184) sorts by `submitted_at` desc and returns the most-recent. Revoked/older ARs remain in the `revoked` bucket of the list endpoint.
- **Root cause**: `concept-doc-not-consulted` (plan + ADR framed UNIQUE semantics at the wrong tier).
