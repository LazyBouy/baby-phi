## Concept alignment
- **Concept doc(s)**: [`concepts/ontology.md`](../../../concepts/ontology.md) §"table-per-node-tier" (persistence ontology) + [`concepts/phi-core-mapping.md`](../../../concepts/phi-core-mapping.md) §Session (wrap boundary)
- **Concept claim (verbatim or close paraphrase)**: Concept docs treat `session` / `loop_record` / `turn` as three distinct node tiers with governance-owned columns layered on top of phi-core's wrapped types. They do not specify migration mechanics (DEFINE TABLE vs DEFINE FIELD); the ontology is about node shape, not SurrealDB DDL order.
- **Contradiction**: None at the concept layer — this drift is about migration tooling semantics (SurrealDB rejects redundant DEFINE TABLE on pre-existing tables), not conceptual model divergence.
- **Classification**: `concept-silent-plan-filled-gap`
- **phi-core leverage status**: `wrap` — the `session.inner` / `turn.inner` columns persist the wrapped phi-core types via `FLEXIBLE TYPE object`. Leverage pattern intact.

## Plan vs. reality
- **Plan said** (plan-archive line 1112, P1 Deliverables item 1): *"DEFINE 3 new session tables: `session`, `loop_record`, `turn` with `FLEXIBLE TYPE object` for the phi-core `inner` field + explicit governance columns."*
- **Reality (shipped state at current HEAD)**: Migration 0001 ([`0001_initial.surql:142-149`](../../../../../../modules/crates/store/migrations/0001_initial.surql#L142-L149)) already shipped `DEFINE TABLE session SCHEMAFULL;` + `DEFINE TABLE turn SCHEMAFULL;` as id-only scaffolds with a pre-existing mandatory `created_at: string` column. Migration 0005 uses `DEFINE FIELD` to layer the M5 governance columns on those scaffolds. Only `loop_record` is a fresh `DEFINE TABLE`. The 0001 `loop` scaffold table stays as a zombie (no writers).
- **Root cause**: `concept-doc-not-consulted` (schema-mechanics layer); the plan wrote "DEFINE 3 new tables" without auditing 0001.
